Part 1 · Section 6 · 15 Points
Mapping the Problem Statement,
Analysis Techniques, and Variables
Choosing the right analytical method before touching any data.
15 pts
Why this matters first: The most common analytical failure isn't a math error — it's choosing the wrong method. A manager who applies logistic regression to a continuous outcome, or runs a simple regression when the data is time-ordered, will produce confident-looking numbers that answer the wrong question. This section teaches you to read a business problem and select the appropriate technique before any analysis begins.

Four diagnostic questions determine which inference technique is appropriate. Work through them in sequence for any new business problem.

  • 1What is the outcome variable (Y)?
    Is it a continuous number (revenue, cost, units sold), a binary yes/no (churned, defaulted, purchased), or a count/proportion observed in categories? The outcome type is the primary filter.
  • 2How many predictor variables (X) do you have?
    One predictor → simple regression. Multiple simultaneous predictors → multiple regression or logistic regression. Knowing which variables matter equally is a key MLR insight: partial effects isolate each driver's contribution while holding all others constant.
  • 3Is the data collected over time in sequence?
    If observations are ordered in time and you expect past values to predict future ones, standard regression violates independence. Use time-series methods (ARIMA, ETS) that explicitly model temporal structure.
  • 4What decision will this analysis support?
    A predictive model (forecast next quarter) requires different validation than an explanatory model (what drives customer CLV). Define the decision before finalizing the technique — this determines which performance metric matters most.

Variable classification determines what transformations and model families are valid. Misclassifying a variable is a fundamental error that can invalidate entire analyses.

⚠ Dummy variable trap
If a categorical variable has k levels (e.g., 4 regions: NE, SE, MW, W), create k−1 dummy variables (3 dummies). Including all k creates perfect multicollinearity — the model cannot be estimated. Always drop one category as the reference group and interpret all other coefficients relative to it.

Apply the framework to three real consulting scenarios. Identify Y, the predictor set, data structure, and correct technique.

Scenario A — Pacific Rim Healthcare
A regional hospital network wants to predict average patient length of stay (LOS in days) using: patient age (years), diagnosis severity score (1–5), insurance type (4 categories), and whether the admission was emergency vs. planned (binary). Data consists of 8,400 individual patient records, cross-sectional.

Y: Continuous (days) · X count: 4 predictors (mixed types) · Structure: Cross-sectional, not time-ordered
✓ Correct technique: Multiple linear regression. Diagnosis severity and insurance type need dummy encoding. Model produces an expected LOS with CI, and partial coefficients showing each factor's independent contribution to hospital resource planning.
Scenario B — CloudPeak SaaS
A SaaS company wants to predict which customers will cancel their subscription next month (cancel = 1, retain = 0) using: tenure in months, number of support tickets in past 30 days, days since last login, plan tier (Basic/Pro), and team seat count.

Y: Binary (0/1) · X count: 5 predictors · Structure: Cross-sectional per customer-month
✓ Correct technique: Logistic regression. Output is a churn probability per customer. Classification threshold is a business decision (not defaulted to 50%) based on the asymmetric cost of missing a churner vs. contacting a loyal customer unnecessarily.
Scenario C — Harvest Foods Distribution
A national grocery distributor needs to forecast monthly produce demand (units) over the next 6 months using 4 years of historical monthly demand data. The series shows clear upward trend and strong Q4 seasonality.

Y: Sequential over time · X: Lagged values of Y; seasonal structure · Structure: Time-ordered monthly data
✓ Correct technique: SARIMA (Seasonal ARIMA). First test stationarity (ADF), difference if needed. Identify AR/MA order via ACF/PACF. Output includes point forecasts and confidence intervals that widen appropriately as the horizon extends.
Question 1 of 2A regional bank wants to predict the dollar amount of annual mortgage default losses using: loan-to-value ratio (%), borrower FICO score, property type (4 types), loan origination year, and local unemployment rate. Which technique is appropriate and why?
ALogistic regression — financial risk modeling always uses binary classification
BMultiple linear regression — the outcome is a continuous dollar amount and multiple predictors are specified simultaneously
CTime-series ARIMA — loan origination year makes this time-ordered data
DBoth logistic and linear regression — run both and choose the one with higher R²
✓ Correct. The outcome is a continuous dollar amount (not a binary yes/no), which immediately rules out logistic regression. While loan origination year is in the dataset, it is used as a predictor variable, not as a sequential time-ordered structure — the observations are individual loans, not a time series of aggregate loss rates. Multiple linear regression is correct: it estimates each predictor's partial effect on dollar losses while holding all other factors constant. Property type (4 categories) requires 3 dummy variables with one category as the reference group. Note: if the question were "will this loan default? (yes/no)," logistic regression would be correct — the outcome type is the decisive factor.
Question 2 of 2Your CFO asks you to predict next quarter's operating expenses for budget planning. You have 5 years of monthly expense data. A colleague suggests multiple linear regression using month number (1–60) as the only predictor. What is the fundamental problem with this approach?
ANo problem — month number is a valid continuous predictor in MLR
BThe model needs more predictors — add headcount and revenue as X variables
CSequential monthly observations are autocorrelated — MLR's independence assumption is violated, and the model ignores seasonal patterns that ARIMA would capture
D60 data points is too few for regression — collect at least 200 observations first
✓ Correct. This scenario has two compounding problems. First, consecutive monthly observations are not independent — this month's expenses are correlated with last month's, violating MLR's independence assumption. Running OLS on autocorrelated data produces biased standard errors and unreliable p-values. Second, a simple linear trend line (month number as X) completely ignores any seasonal structure in expenses — for example, if Q4 consistently runs 20% higher due to year-end vendor contracts, the linear model has no mechanism to capture that. ARIMA (specifically SARIMA for seasonal data) is designed for exactly this problem: it models the temporal autocorrelation structure and seasonal periodicity jointly. The correct approach: test stationarity with ADF, examine ACF/PACF for lag structure, identify seasonal period (12 for monthly), and fit a SARIMA model that produces forecasts with appropriate confidence intervals.
Part 2 · Section 6 · 25 Points
Multiple Linear Regression Analysis
Isolating the independent effect of each driver on a continuous business outcome.
25 pts
Business scenario: You are VP of Revenue Strategy at Precision Retail Co., a mid-market chain with 48 stores. Your analytics team modeled weekly store revenue ($) using 104 weeks of data. Predictors include advertising spend ($), discount depth (%), foot traffic (store visits), regional unemployment rate (%), and a binary holiday-week indicator. The model uses n = 4,992 store-weeks.

MLR fits a hyperplane through n-dimensional space that minimizes the sum of squared residuals. Each coefficient β represents a partial effect — the change in Y for a one-unit increase in Xᵢ, holding all other predictors constant.

Ŷ = β₀ + β₁X₁ + β₂X₂ + ... + βₖXₖ + ε β₀ = intercept (predicted Y when all X = 0) βᵢ = partial slope: ΔY per unit ΔXᵢ, holding all Xⱼ constant ε = residual error (unexplained variation) H₀: βᵢ = 0 # Variable i has no effect on Y Hₐ: βᵢ ≠ 0 # Variable i has a significant effect on Y # Estimation via OLS — minimizes Σ(Yᵢ − Ŷᵢ)² from sklearn.linear_model import LinearRegression model.fit(X_train, y_train) coefficients = model.coef_ r_squared = model.score(X_test, y_test)
Why "partial" effects matter in management
In a simple regression of revenue on ad spend, the β coefficient absorbs all other things correlated with spending — seasonality, competitor behavior, economic conditions. In MLR that includes those variables, βad spend becomes a cleaner, more defensible estimate of advertising's independent contribution. This is ceteris paribus reasoning — the statistical equivalent of holding everything else constant while you test one lever.

The LINE assumptions must be assessed before trusting coefficients, standard errors, or p-values. Violations change what the output means — they don't always invalidate the model, but they must be disclosed.

✓ Linearity — met
Residuals vs. fitted plot shows random scatter around zero. No systematic curvature. If violated: add polynomial terms (X²) or transform Y with log/sqrt.
✓ Independence — met
Durbin-Watson = 1.94 (range 0–4; near 2 = no autocorrelation). Satisfied here because observations are store-weeks drawn independently across 48 locations.
⚠ Normality — mild concern
Q-Q plot shows minor right-skew in extreme residual tails. With n = 4,992, CLT ensures inference is still valid. Flag in methods section; log-transform Y if RMSE does not improve.
✗ Equal variance — violated
Breusch-Pagan test: p = 0.003. Variance increases with fitted values (heteroskedasticity). Fix: use robust standard errors (HC3) — standard SEs and CIs are understated. Always report robust SEs when BP p < 0.05.
VIF (Variance Inflation Factor) — multicollinearity check
Ad spend VIF = 3.2; Foot traffic VIF = 2.9; all others < 2.0. VIF < 5 is generally acceptable. VIF > 10 indicates severe multicollinearity — individual coefficients become unstable and uninterpretable even if R² remains high. Always check VIF before presenting partial coefficients to leadership.

Question 1 of 2A board member says: "Our ad spend coefficient is $4.10 — so if we triple our spend from $20k to $60k per week, revenue will increase by exactly $164,000." What is the correct analytical response?
AAgree — the linear model predicts this precisely and it is reliable across all ranges
BThe model supports that directional prediction within the observed data range, but extrapolating to $60k introduces substantial uncertainty — the linear relationship may not hold at that spend level, and diminishing returns are well-documented in advertising science
CThe model is useless for budget decisions — only randomized experiments can guide ad spend
DThe model already accounts for diminishing returns through its quadratic terms
✓ Correct. This is the extrapolation trap. The β coefficient is a locally valid estimate within the range of observed training data. If stores historically spent between $5k–$35k/week, the linear assumption at $60k is an out-of-sample projection, not an interpolation. Advertising has well-documented diminishing returns at high spend levels. The board member is making a reasonable planning estimate directionally, but the analyst's job is to communicate the widening uncertainty band and recommend a controlled spend experiment at higher budget levels before committing. The CI of [$3.36, $4.84] already acknowledges parameter uncertainty — it does not capture structural model uncertainty from extrapolation.
Question 2 of 2The model's adjusted R² is 0.849. A colleague proposes adding 8 more demographic variables to push R² above 0.90. What is the statistically correct response?
AAdd all 8 — higher R² always means a better model
BEvaluate each variable on theoretical justification and check whether it improves adjusted R² and holdout performance — adding noise variables inflates R² while degrading generalization
CAn adjusted R² of 0.849 is unacceptable — all business models should exceed 0.95
DUse stepwise regression to automatically select all variables that improve R²
✓ Correct. R² never decreases when you add variables — even random noise will marginally increase it. Adjusted R² penalizes for predictor count and will fall if a new variable contributes less explanatory power than it costs in degrees of freedom. The correct test for adding a variable: (1) Is there a theoretical reason it should predict revenue? (2) Does it improve adjusted R² and cross-validated test performance? (3) Does it change existing coefficients in ways that suggest multicollinearity? Stepwise regression, while computationally convenient, produces models optimized for one training dataset that generalize poorly. Modern best practice is theory-guided variable selection with holdout validation.

Logistic Regression Analysis
Predicting binary outcomes and optimizing classification thresholds for real cost structures.
20 pts
Business scenario: You are Chief Customer Officer at CloudPeak SaaS (140,000 B2B subscribers; $89/mo Basic, $220/mo Pro). Monthly churn averages 3.8%. Your retention team can proactively contact 500 customers/month. A false negative (missed churner) costs ~$1,200 in LTV. A false positive (contacting a loyal customer) costs $30. Your logistic regression model was trained on n = 28,400 customer-months.

Logistic regression maps predictors to a probability in (0, 1) via the logistic (sigmoid) function. Unlike linear regression applied to a 0/1 outcome, predictions can never exceed 1.0 or fall below 0 — critical when outputs are used as probabilities for business decisions.

# The logistic model — log-odds form log(p / 1−p) = β₀ + β₁X₁ + β₂X₂ + ... + βₖXₖ # Converting log-odds to probability p = 1 / (1 + e⁻ᶻ) where z = β₀ + ΣβᵢXᵢ # Odds ratio for variable Xᵢ OR = e^βᵢ # OR > 1: each unit increase in Xᵢ multiplies churn odds by OR # OR < 1: each unit increase in Xᵢ reduces churn odds by (1−OR)×100% from sklearn.linear_model import LogisticRegression model.fit(X_train, y_train) probs = model.predict_proba(X_test)[:, 1] # churn probabilities preds = (probs >= threshold).astype(int) # classify at chosen threshold
⚠ Why not use linear regression for binary outcomes?
OLS applied to a 0/1 outcome (linear probability model) produces predictions that can exceed 1.0 or fall below 0.0 — mathematically nonsensical for probabilities. Logistic regression's sigmoid transformation ensures all outputs stay in (0, 1). Additionally, the residuals of an LPM are heteroskedastic by construction, inflating standard errors and undermining inference.

Question 1 of 2Your logistic model has AUC-ROC = 0.847. A junior analyst tells the executive team "our model is 84.7% accurate." What is the most precise correction?
AAgree — AUC-ROC directly measures classification accuracy
BAUC-ROC measures rank-ordering ability across all thresholds — it is the probability that the model ranks a random churner above a random non-churner. Accuracy depends on the specific threshold chosen and should be reported from the confusion matrix.
CThe model needs AUC ≥ 0.90 before it can be presented to leadership
DAUC-ROC is irrelevant — only the confusion matrix matters for business decisions
✓ Correct. AUC-ROC (Area Under the Receiver Operating Characteristic Curve) is a threshold-independent metric. It measures discriminatory power: if you randomly draw one churner and one non-churner from the test set, an AUC of 0.847 means the model assigns the churner a higher probability 84.7% of the time. It is not an accuracy percentage. Accuracy at a given threshold is computed from the confusion matrix and will be different from the AUC. For the executive briefing: translate AUC as "the model correctly ranks churners above non-churners 85% of the time when comparing pairs" — that's the intuitive, decision-relevant framing.
Question 2 of 2A customer has an OR of 3.136 for "payment failed." The CMO interprets this as "customers with payment failures are 213.6% more likely to churn." Is this correct?
AYes — OR = 3.136 means 213.6% higher relative likelihood of churning
BNo — the OR of 3.136 means customers are 3.136 percentage points more likely to churn
CPartially. OR = 3.136 means the odds of churning are 3.136× higher (213.6% higher odds), not 213.6% higher probability. For rare outcomes like churn, OR approximates relative risk — but the distinction matters for accurate executive communication.
DNo — odds ratios cannot be interpreted as relative probabilities under any circumstances
✓ Correct. Odds ratios and relative risk are distinct. OR = 3.136 means the odds of churning (p/1-p) are 3.136× higher for payment-failure customers. For rare outcomes (base rate ~3.8%), the OR approximates relative risk closely — so "roughly 3× more likely to churn" is a defensible simplification for the CMO. However, saying "213.6% more likely" conflates odds with probability. The technically precise statement: "Customers with a payment failure have 3.1× higher odds of churning in the following month, holding all other factors constant." For non-technical audiences, a simpler framing works: "A payment failure is the single strongest predictor of imminent churn — these customers should be the first contact priority regardless of threshold."
Part 4 · Section 6 · 20 Points
Time-Series Analysis & Forecasting: Introduction
Modeling sequential data, testing stationarity, and producing actionable demand forecasts.
20 pts
Business scenario: You lead supply chain analytics at Harvest Foods Co., a national grocery distributor. You have 4 years (48 months) of historical produce demand data showing a consistent upward trend (+2.4%/quarter) and strong Q4 seasonality (±21%). You must forecast the next 6 months to drive procurement contracts and prevent both overstock waste and stockout lost sales.

Standard regression assumes observations are independent. In time-series data, today's value is correlated with yesterday's — violating independence. ARIMA models this autocorrelation structure explicitly, treating past values of Y as predictors of future Y.

# ARIMA(p, d, q) model specification Yₜ = c + φ₁Yₜ₋₁ + ... + φₚYₜ₋ₚ + εₜ − θ₁εₜ₋₁ − ... − θqεₜ₋q p = autoregressive order (# lagged Y values included) d = differencing order (# times series was differenced) q = moving average order (# lagged error terms included) # For seasonal data: SARIMA(p,d,q)(P,D,Q)₁₂ H₀: unit root present → series is non-stationary (ADF test) Hₐ: no unit root → series is stationary from statsmodels.tsa.statespace.sarimax import SARIMAX from statsmodels.tsa.stattools import adfuller adf_result = adfuller(demand_series) # p > 0.05 → difference needed
The spurious regression trap
If you regress two non-stationary (trending) time series against each other with OLS, you will almost always find high R² and significant p-values — even when the variables are completely unrelated. This is spurious regression. A well-documented example: US cheese consumption and deaths by bedsheet tangling correlate at R² = 0.95 over 10 years. Stationarity testing prevents this fundamental error.

Before fitting a model, decompose the series into its structural components. This builds intuition about what the model must capture and reveals whether trend and seasonality are stable enough to extrapolate.

Stationarity means the series has a constant mean, constant variance, and no systematic trend over time. ARIMA requires stationarity — or differenced-to-stationarity data.

  • 1Run the Augmented Dickey-Fuller (ADF) test
    H₀ = unit root present (non-stationary). If p ≥ 0.05, fail to reject — the series is non-stationary. Do not proceed to ARIMA without differencing.
  • 2Apply first differencing if needed
    Compute Δyₜ = yₜ − yₜ₋₁. This removes a linear trend. The d parameter in ARIMA(p,d,q) records how many times you differenced. d = 1 handles most business series.
  • 3Re-run ADF on the differenced series
    If p < 0.05 after first differencing: stationarity achieved. Set d = 1 and proceed. If still non-stationary, difference again (d = 2) — rare for standard business data.
  • 4Check for seasonal differencing
    If seasonal pattern persists after regular differencing, apply seasonal differencing: Δ₁₂yₜ = yₜ − yₜ₋₁₂ (for monthly data with annual seasonality). The D parameter in SARIMA records this.

Reading forecasts for procurement decisions
Use the upper bound of the CI for inventory procurement, not the point estimate. Stockout costs (lost sales + customer attrition + contract penalties) typically exceed overstock costs (storage + markdown) in grocery supply chains. The CI widens as the horizon extends — a 2-month forecast may have ±6% uncertainty while a 9-month forecast has ±18%. Communicate the CI range to the COO alongside the point estimate; leaders who see only one number assume more certainty than exists.

Question 1 of 2ADF test on monthly revenue returns p = 0.38. You difference the series once. The new ADF returns p = 0.004. What does this mean and how should you proceed?
ABoth p-values confirm stationarity — proceed with the original series
BOriginal series had a unit root (non-stationary); one round of differencing achieved stationarity; set d = 1 and proceed with ARIMA(p,1,q)
CThe series is still non-stationary — apply a log transformation and re-test
DUse the undifferenced series but apply first-order smoothing as compensation
✓ Correct. ADF H₀ = unit root present (non-stationary). p = 0.38 → fail to reject H₀ → non-stationary. After first differencing, p = 0.004 → reject H₀ → differenced series is stationary. This tells you d = 1 in ARIMA(p,1,q). In practical terms, your revenue series has a stochastic trend — a random walk component that differencing removes. This is the standard finding for most business revenue and demand series. Use ACF/PACF on the differenced series to determine p and q. Had the series remained non-stationary after d=1, a second difference (d=2) would be applied — but this is uncommon in practice and often signals a measurement or structural issue in the data.
Question 2 of 2Your Q2 demand forecast is 4,200 units with an 80% CI of [3,700, 4,900]. Your COO asks whether to commit to a $4.5M procurement contract (requiring ~4,500 units). How do you advise?
ACommit — 4,500 is within the 80% CI so it's a safe procurement level
BPresent the full CI and frame the decision around stockout cost vs. overstock cost at 4,500 units — the CI alone cannot determine the right procurement level without a cost model
CDo not commit — the point estimate is 4,200 and procurement above that level is statistically unjustified
DUse the lower bound (3,700) — conservative planning minimizes all risk
✓ Correct. A forecast without a cost structure is an incomplete recommendation. The right framing: "If we commit to 4,500 units and demand comes in at 3,700 (lower bound), we carry 800 units in overstock — what does that cost in storage and markdown? If we commit to 4,200 and demand hits 4,900, we're 700 units short — what does that cost in lost sales and customer attrition?" The CI tells you the range your procurement decision must survive. Cost asymmetry — not the CI alone — drives the optimal procurement level. In most grocery supply chains, stockout costs exceed overstock costs, so skewing toward the upper CI bound is rational. Always pair the forecast output with explicit cost modeling before presenting a procurement recommendation.
Part 5 · Section 6 · 15 Points
Validating Prediction Models
Ensuring your model generalizes to new data before committing organizational resources to its recommendations.
15 pts
Business scenario: You are VP of Analytics at Summit Regional Health, a health insurer. Your team built a regression model predicting annual patient cost ($) using age, BMI, smoking status, diagnosis category, and plan type. Before pricing 200,000 policies with this model, you must validate it will not systematically misprice due to overfitting.

A model that performs well on training data may perform far worse on new data. This gap — between training performance and real-world performance — is the central risk in any analytical deployment. As a manager, closing this gap before committing resources is your responsibility.

Data leakage — the #1 cause of models that appear to work until deployed
Data leakage occurs when information from the future or from the test set bleeds into model training. Example: a churn model trained on data that includes whether the customer had already filed a cancellation request (a future-derived feature). It will achieve near-perfect training accuracy and collapse in production. Leakage produces models that look spectacular on paper and fail catastrophically in practice. Check every feature for temporal precedence: can this feature be known at prediction time?
  • 1Split data before any preprocessing
    The train/test split must happen before imputation, scaling, or feature engineering. Fitting a scaler on combined train+test data leaks test statistics into training — a subtle but common error.
  • 2Never look at test data until final evaluation
    Iterating model design based on test performance converts your holdout into another training signal. Use a validation set for tuning; reserve the test set for a single final evaluation that represents real-world performance.
  • 3Report test performance, not training performance
    Any model report presented to executive leadership must prominently feature holdout or cross-validated performance. Training R² alone is a misleading, incomplete metric. If someone shows you only training performance, ask for holdout immediately.

Match validation rigor to the stakes of the decision. A dashboard metric for internal use has different requirements than a pricing model applied to 200,000 policies.

Level 1
Train/test split
80/20 holdout. Fast, minimum viable. Use for low-stakes internal tools.
Level 2
K-fold CV
5 or 10 folds. Better variance estimate. Standard for most production models.
Level 3
Nested CV
CV within CV. Unbiased when tuning hyperparameters simultaneously with evaluation.
Level 4
Temporal holdout
For time-series: always train on past, test on future. Never shuffle time-series data randomly.
⚠ Class imbalance trap — the accuracy illusion
A churn model on a dataset with 3.8% churn rate can achieve 96.2% accuracy by predicting "no churn" for every single customer — a completely useless model. Always report precision, recall, and F1 on the minority class for imbalanced datasets. Overall accuracy is meaningless when class proportions are unequal. This mistake is made regularly by analysts presenting results to non-technical leadership.

Question 1 of 2A model predicting high-risk insurance applicants achieves: Train accuracy = 94.2%, Test accuracy = 91.8%, AUC-ROC = 0.73. Base rate of high-risk applicants = 6%. What is your primary analytical concern?
AThe 2.4% train-test gap indicates severe overfitting
BAUC of 0.73 is too low — models must exceed 0.85 before any deployment
CWith 6% base rate, predicting "low risk" for every applicant achieves 94% accuracy — the 91.8% test accuracy may not meaningfully exceed this naïve baseline, making accuracy a misleading metric here
DThe model is ready — 91.8% holdout accuracy is strong for any high-stakes domain
✓ Correct. This is the class imbalance trap. With 6% base rate, a naïve classifier predicting "low risk" for everyone achieves 94% accuracy — outperforming this model's 91.8%! The accuracy metric is measuring near-zero signal. The AUC of 0.73 is more informative — it shows some discriminatory power — but the confusion matrix is what matters most. You need to see: of the 6% truly high-risk applicants, how many does the model correctly identify (recall)? If recall on high-risk = 40%, the model misses 60% of the applicants it was built to catch — a regulatory and financial liability. Always compute precision/recall on the minority class, never rely on overall accuracy with imbalanced outcomes.
Question 2 of 2Your Summit Health pricing model shows Train R² = 0.94, Test R² = 0.57. You plan to price 200,000 policies next quarter. What is the correct recommendation?
AProceed — the model explains 94% of variance and the test performance will improve with more data
BAverage the two R² values (0.755) and report that as the model's expected real-world performance
CFlag as severe overfitting — the model is not deployment-ready; pursue regularization, feature reduction, or leakage investigation before any policy pricing
DRe-run the model including test data in training to improve the test R²
✓ Correct. A gap of 0.37 between train and test R² is a textbook case of severe overfitting. The model has learned noise patterns in the training data that do not exist in the real population. Deploying it to price 200,000 health policies would create systematic mispricings — some policies underpriced (adverse selection risk) and others overpriced (competitive loss). The correct next steps: (1) check for data leakage, (2) apply L1/L2 regularization (Ridge/Lasso), (3) reduce feature count and re-evaluate via cross-validation, (4) consider gathering more training data. Option D — including test data in training — is never acceptable. It eliminates the holdout set entirely and makes final performance evaluation impossible. The test set must remain untouched until the model is finalized.

🚩
Red flag 1 — Only training performance is reported
Any model report showing only in-sample R² or accuracy has not been validated. Ask immediately: "What is the holdout or cross-validated performance?" This is non-negotiable before deployment.
🚩
Red flag 2 — Accuracy on an imbalanced dataset
"97% accurate" on a fraud dataset where 97% of transactions are legitimate is a null model. Always ask for precision and recall on the minority class. Overall accuracy is misleading when classes are unbalanced.
🚩
Red flag 3 — Single-number forecasts without uncertainty
A forecast without a confidence interval is a false precision claim. Uncertainty compounds over longer horizons. Stakeholders who receive only point estimates tend to over-rely on them.
🚩
Red flag 4 — Statistical significance confused with practical importance
A β with p < 0.001 on a 50,000-row dataset can represent an effect too tiny to matter operationally. Always ask: "What is the effect size in business units, and does it justify the cost of acting on it?" p-value and effect size are independent.
🚩
Red flag 5 — Correlation presented as causation
Regression identifies associations. Claiming that X "causes" Y requires experimental design (randomized control trial) or a credible natural experiment. A positive coefficient on advertising spend does not prove advertising caused the revenue lift — confounds remain until controlled.
Question 1 of 3A model report states: "Our logistic regression achieves 96% accuracy on the held-out test set. The model is ready for deployment." The dataset has 4% base rate of positive outcomes. What should you conclude?
AThe model is ready — 96% holdout accuracy is excellent for any application
BThis report is incomplete — a naïve model predicting "negative" for all instances achieves 96% accuracy; AUC, precision, and recall on the positive class are required before any deployment decision
CThe model likely overfits — high accuracy on holdout data is a sign of data leakage
DThe model needs more training data before the accuracy figure is meaningful
✓ Correct. With a 4% positive base rate, a model that predicts "negative" for every single observation achieves 96% accuracy without learning anything. The accuracy metric provides zero information about model quality here. The required metrics: (1) AUC-ROC — discriminatory power independent of threshold, (2) Precision on positives — what fraction of predicted positives are real, (3) Recall on positives — what fraction of all true positives did the model catch. A model with 96% accuracy but 20% recall is missing 80% of the outcomes it was built to find — that's not "ready for deployment," it's a false-negative machine.
Question 2 of 3An executive shows you a regression analysis showing that companies with more social media followers have higher revenue, with β = +$450 per follower, R² = 0.71, p < 0.001. She concludes: "We should invest $2M in a follower acquisition campaign." What is the analytical concern?
AThe R² of 0.71 is too low to support a $2M decision
BThe p-value is significant so the coefficient is reliable for budget planning
CThis is an observational association, not a causal relationship — large companies have both more revenue and more followers; the coefficient may reflect size confounding rather than a real lever. A causal claim requires experimental or quasi-experimental evidence.
DThe model needs additional variables like industry and geography before the β can be trusted
✓ Correct. This is the correlation-causation trap at its most common in business settings. Larger companies have both more revenue and more social media followers — both are likely caused by a third variable (company size, brand maturity, marketing budget). The regression coefficient captures this confounded association. Buying followers does not make you a larger company; it just changes one symptom without addressing the underlying driver. The $2M decision requires causal evidence: a randomized experiment where follower count is randomly varied while controlling for size, or a natural experiment (e.g., a platform policy change that arbitrarily affected follower counts for some companies but not others). Observational regression alone never justifies a causal resource allocation claim.
Question 3 of 3You are reviewing a demand forecasting model for a retail chain. The analyst reports MAPE = 6.2% on the training data. The model will be used to commit to 12-month supply contracts. What validation gap must you close before approving the model for this use?
ANothing — MAPE of 6.2% is excellent and training performance is sufficient for long-range supply commitments
BConvert MAPE to RMSE — RMSE is the appropriate metric for supply chain applications
CThe MAPE is in-sample only — you must evaluate holdout MAPE on future periods (temporal holdout) to know whether the model generalizes forward in time before committing 12-month contracts to its forecasts
DExpand the training data to the full historical period and re-report MAPE before committing
✓ Correct. Training MAPE tells you how well the model fits data it has already seen — a prerequisite but not a sufficient condition for deployment. For a time-series model, the only valid validation is a temporal holdout: train on early periods, test on the most recent periods you actually have (e.g., hold out the last 6–12 months). This simulates exactly the real-world task — predicting future periods from past ones. Shuffled cross-validation is inappropriate for time-series because it leaks future information into training folds. A 6.2% training MAPE that becomes 18% on a 6-month temporal holdout reveals a model that has overfit to historical noise patterns rather than learning the true demand structure. Only temporal holdout MAPE is relevant for approving 12-month supply contract commitments.
